## 第一章：系统概述

### 1.1 系统简介

OpsMonitor 是一套面向团队和中小型项目的全栈服务监控系统，采用现代化架构设计，提供全方位的服务健康监控、依赖关系可视化与智能告警能力。系统同时支持本地和远程（SSH）双模式部署，适用于多项目的生产环境运维场景。

**核心功能亮点**：

| 功能 | 说明 |
|------|------|
| 🏗️ **三层架构管理** | 项目 → 主机 → 服务的层级化组织方式 |
| 🔍 **多种健康检查** | TCP、HTTP/HTTPS、自定义脚本、文件监控（本地及 SSH 远程） |
| 📊 **依赖可视化** | 交互式服务拓扑图，支持影响分析与依赖追踪 |
| 🔔 **智能告警通知** | 支持邮件（SMTP）和 Teams Webhook，可自定义告警规则与阈值 |
| 📈 **Grafana 集成** | 嵌入外部 Grafana 仪表板，统一展示 |
| 🔐 **安全管理** | AccessKey、SSL 证书、密码的过期监控与提醒 |
| 🔄 **动态调度** | 灵活的定时检查配置，支持时间范围限制 |
| 🌍 **多语言支持** | 中英文界面一键切换，告警通知语言自动同步 |

### 1.2 系统架构

OpsMonitor 采用前后端分离架构，配合嵌入式数据库，实现开箱即用：

```
┌─────────────────────────────────────────────────────────────────┐
│                       OpsMonitor                                │
├──────────────────────────────┬──────────────────────────────────┤
│   前端 (Vue 3 + Element Plus)│   后端 (Node.js + Express)      │
│      端口: 5173              │      端口: 3000                  │
├──────────────────────────────┴──────────────────────────────────┤
│                     SQLite 嵌入式数据库                          │
└─────────────────────────────────────────────────────────────────┘
```

**三层管理架构**：

```
项目（Project）
  └── 主机（Host）          ← 配置 SSH 或 Local 连接
        └── 服务（Service） ← 配置健康检查类型和调度
```

- **项目**：业务或环境的逻辑分组，例如"生产环境"、"开发环境"
- **主机**：实际的服务器或目标机器，包含连接配置
- **服务**：具体的监控项，每个服务对应一种健康检查

### 1.3 用户角色与权限

系统提供两种用户角色：

| 角色 | 英文标识 | 权限范围 |
|------|---------|---------|
| **管理员** | Admin | 完整权限：项目/主机/服务的增删改查、告警与安全配置、用户管理、系统设置 |
| **观察者** | Viewer | 只读权限：查看监控数据、仪表盘；可手动触发健康检查，不可修改任何配置 |

> 新注册账号默认角色为 **Viewer**，需由管理员手动升级为 Admin。

### 1.4 技术栈概览

| 层次 | 技术组件 |
|------|---------|
| **前端** | Vue 3 · TypeScript · Element Plus · ECharts · Pinia · Axios · vue-i18n |
| **后端** | Node.js · Express · TypeScript · JWT · node-cron |
| **数据库** | SQLite（嵌入式，无需独立部署） |
| **远程通信** | ssh2（SSH 连接池） |
| **容器化** | Docker + Docker Compose |

## 第二章：部署与安装

### 2.1 环境要求

| 组件 | 最低版本 | 说明 |
|------|---------|------|
| Node.js | 18.0+ | 本地开发必需 |
| npm | 随 Node.js | 包管理器 |
| Docker | 20.10+ | 生产部署时使用 |
| Docker Compose | v2.0+ | 生产部署时使用 |
| 操作系统 | Windows / Linux / macOS | 均支持 |

### 2.2 本地开发启动

**方式一：使用快速启动脚本（推荐）**

```powershell
# Windows (PowerShell)
cd OpsMonitor
.\dev.ps1
```

```bash
# Linux / macOS
cd OpsMonitor
./dev.sh
```

**方式二：手动启动**

```bash
# 终端 1：启动后端 (端口 3000)
cd OpsMonitor/backend
npm install
npm run dev

# 终端 2：启动前端 (端口 5173)
cd OpsMonitor/frontend
npm install
npm run dev
```

### 2.3 生产环境 Docker 部署

```bash
cd OpsMonitor
docker compose up -d
```

Docker 将自动完成前后端构建与启动，**无需手动执行 `npm run build`**。

| 服务 | 端口 | 说明 |
|------|------|------|
| 前端 | 5173 | Nginx 静态服务 + API 代理 |
| 后端 | 3000 | Express API 服务 |
| 数据库 | — | SQLite 文件，持久化到 `/opt/OpsMonitor/backend/data` |

### 2.4 环境变量配置

**所有部署方式均无需创建任何配置文件**，项目已内置全部默认值，开箱即用。

仅当需要覆盖默认值时，可在 `backend/` 目录创建 `.env` 文件：

```env
NODE_ENV=development
PORT=3000
DB_PATH=./data/monitor.db
JWT_SECRET=your-secret-key-here
```

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `NODE_ENV` | `production` | 运行模式 |
| `PORT` | `3000` | 后端监听端口 |
| `DB_PATH` | `/app/data/monitor.db` | SQLite 数据库文件路径 |
| `JWT_SECRET` | `opsmonitor-secret-key-change-in-production` | JWT 签名密钥，**生产环境务必替换** |

### 2.5 访问地址

| 地址 | 说明 |
|------|------|
| `http://localhost:5173` | 前端用户界面 |
| `http://localhost:3000/health` | 后端健康检查接口 |
| `http://localhost:3000/api` | 后端 API 前缀（需跟具体路径） |

## 第五章：概览仪表盘

**页面路径**：`/`（系统首页）

概览仪表盘是系统的全局监控入口，汇总展示所有服务的实时健康状态。

![概览仪表盘](images/ch02-dashboard-overview.png)

### 5.1 统计卡片

页面顶部展示四个统计卡片：

| 卡片 | 说明 |
|------|------|
| **Total Services** | 当前项目（或全部项目）的服务总数 |
| **Healthy** | 状态为 UP 的服务数量 |
| **Warning** | 状态为 WARNING 的服务数量 |
| **Down** | 状态为 DOWN/UNKNOWN 的服务数量 |

每张卡片下方附有进度条，直观展示各状态的占比。

### 5.2 图表展示

统计卡片下方展示两个图表卡片以及系统概览区域：

| 图表 | 说明 |
|------|------|
| **状态分布饼图** | 展示 UP / WARNING / DOWN / UNKNOWN 各状态的服务占比 |
| **项目服务柱状图** | 按项目分组展示各项目的服务数量 |
| **系统概览** | 项目数、主机数、依赖数、Grafana 仪表板数、安全配置数、问题数的快速统计 |

### 5.3 自动刷新与手动刷新

页面右上角提供刷新控制区域：

| 控件 | 说明 |
|------|------|
| **Auto / Manual 开关** | 开启/关闭自动刷新 |
| **间隔下拉菜单** | 可选 15秒 / 30秒 / 1分钟 / 5分钟 |
| **倒计时进度条** | 展示距下次刷新的剩余时间 |
| **Refresh 按钮** | 立即手动刷新 |

### 5.4 按项目筛选

通过页面**左侧导航菜单顶部的项目选择器**，可切换查看特定项目的监控数据。选择项目后，仪表盘上的所有统计数据将自动过滤为该项目范围内的数据。

## 第六章：项目管理

**页面路径**：`/projects`

**导航入口**：左侧导航栏 **INFRASTRUCTURE** 组 → **Projects**

项目是服务的逻辑分组，用于组织和隔离不同业务或环境的监控服务。

![项目列表](images/ch03-projects-list.png)

### 6.1 创建项目

有三种方式可以创建新项目：

**方式一：通过 Projects 页面顶部按钮（推荐）**

1. 点击左侧导航栏 **INFRASTRUCTURE → Projects** 进入项目列表页
2. 点击页面右上角 **New Project** 按钮
3. 在弹出对话框中填写：

![新建项目对话框](images/ch03-projects-add-dialog.png)

   - **项目名称**（必填）：如 "生产环境"、"测试环境"
   - **描述**（选填）：项目的详细说明
   - **颜色主题**：为项目卡片选择显示颜色
4. 点击 **Create Project** 保存

**方式二：通过项目列表页的添加卡片**

在项目列表的最后一格有一张 **Add New Project** 卡片，直接点击即可触发新建对话框（与方式一效果相同）。

**方式三：通过侧边栏 Current Project 下拉选择器**

1. 点击左侧导航菜单顶部的 **Current Project** 下拉选择器
2. 在展开的下拉菜单中点击 **New Project**

![项目选择器下拉菜单](images/ch03-project-selector-dropdown.png)

3. 填写项目信息后点击 **Create Project** 保存

### 6.2 编辑项目

1. 鼠标悬停在项目卡片上，点击右上角 **"..."** 菜单
2. 选择 **Edit**
3. 修改项目信息后保存

### 6.3 删除项目

1. 点击项目卡片右上角 **"..."** 菜单
2. 选择 **Delete**
3. 在确认弹窗中确认删除

> ⚠️ 删除项目**不会**删除项目内关联的主机和服务，它们会变为未分配状态。

### 6.4 项目切换与数据过滤

- 点击任意项目卡片，可将该项目设为当前上下文
- 切换后，仪表盘、服务列表等页面将自动过滤显示该项目的数据
- 左侧侧边栏顶部的 **Current Project** 下拉选择器也可快速切换

**Current Project 下拉选择器说明：**

| 选项 | 说明 |
|------|------|
| **All Projects** | 切换至全局视图，显示所有项目数据 |
| 各项目名称 | 切换至该项目上下文，右侧彩色小圆点表示项目健康状态（🟢 健康 / 🟡 警告 / 🔴 故障） |
| **New Project** | 跳转到 Projects 页面并弹出新建对话框 |
| **Manage Projects** | 跳转到 Projects 页面进行项目管理 |

## 第七章：主机管理

**页面路径**：`/hosts`

主机是监控目标的物理/虚拟载体，OpsMonitor 支持两种连接模式：
- **SSH 模式**：用于远程服务器，通过 SSH 执行检查命令
- **Local 模式**：用于本地网络可达的目标，通过直接 TCP/HTTP 访问进行检查

![主机卡片视图](images/ch04-hosts-card-view.png)

### 7.1 视图模式

| 模式 | 说明 |
|------|------|
| **卡片视图** | 以卡片形式展示主机，直观显示连接状态和服务健康统计 |
| **表格视图** | 以表格形式展示，便于对比查看多台主机的信息 |

点击页面右上角的视图切换图标即可在两种模式间切换。

### 7.2 添加主机

点击 **Add Host** 按钮，打开添加主机对话框：

![添加主机对话框](images/ch04-hosts-add-dialog.png)

**基本信息**：

| 字段 | 必填 | 说明 |
|------|------|------|
| Host Name | ✅ | 主机的显示名称，如 `prod-web-01` |
| IP / Hostname | ✅ | 服务器的 IP 地址或域名 |
| Project | — | 关联的项目（可选） |

**连接类型**：

选择 **SSH** 或 **Local**：

- 选择 **Local**：支持基本的网络可达性检查（TCP / HTTP / HTTPS）
- 选择 **SSH**：支持远程命令执行、文件检查等高级功能

**SSH 配置**（选择 SSH 时显示）：

| 字段 | 说明 |
|------|------|
| SSH Port | SSH 端口，默认 22 |
| Username | SSH 登录用户名 |
| Auth Type | 认证方式：Password（密码）或 Private Key（私钥） |
| Password / Private Key | 对应的认证凭据 |
| Proxy Host | 跳板机主机名（可选，用于跳板机场景） |

**高级参数**（可折叠展开）：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| Max Retries | 3 | 连接失败后的重试次数（1-10） |
| Retry Delay | 1s | 重试间隔时长（1-10 秒） |
| Connection Timeout | 10s | SSH 握手超时时间（5-60 秒） |
| Command Timeout | 30s | 远程命令执行超时时间（10-300 秒） |

**监控参数**：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| Host Check Interval | — | 主机连通性检查间隔（60-3600 秒），SSH 模式通过 SSH 连接检测，Local 模式通过 Ping 检测 |

**其他字段**：

| 字段 | 说明 |
|------|------|
| Description | 主机描述信息（选填） |
| Tags | 标签，用于主机分类和筛选 |

### 7.3 跳板机（Proxy Host）配置

当目标服务器无法直接访问，需通过跳板机中转时：

1. 在 **Proxy Host** 字段填写跳板机主机名（需已在系统中将该跳板机作为主机添加）
2. 系统将通过 SSH 隧道经跳板机连接到目标服务器

### 7.4 测试连接

在添加/编辑主机对话框中提供 **Connection Test** 功能：
- **SSH 模式**：点击 **🔌 Test SSH Connection** 按钮，系统将尝试建立 SSH 连接并返回结果
- **Local 模式**：点击 **📡 Test Ping** 按钮，系统将执行 Ping 测试

> 连接测试通常需要 5-15 秒，请耐心等待。

### 7.5 主机状态说明

| 状态标签 | 颜色 | 含义 |
|---------|------|------|

| CONN FAIL | 🔴 红色 | 最近一次连接测试失败 |
| SVC OK | 🟢 绿色 | 该主机下所有服务均正常 |
| SVC WARN | 🟠 橙色 | 该主机下有服务处于警告状态 |
| SVC ERROR | 🔴 红色 | 该主机下有服务处于故障状态 |

## 第八章：服务管理

**页面路径**：`/services`

服务是 OpsMonitor 中监控的最小单元，每个服务对应一个具体的健康检查目标。

![服务列表](images/ch05-services-list.png)

### 8.1 服务的概念

| 概念 | 说明 |
|------|------|
| **服务（Service）** | 一个健康检查项，关联到特定主机上的一种检查任务 |
| **检查类型** | 定义如何检查：TCP 连通性、HTTP 响应、文件存在性等 |
| **风险级别** | 标记服务的重要程度：Low / Medium / High / Critical |
| **调度** | 定义检查频率和时间范围 |

### 8.2 使用向导添加服务

点击 **Add Service** 按钮，启动添加服务向导（分多步引导完成配置）：

**Step 1 — 基本信息**

![添加服务向导-步骤1](images/ch05-add-service-wizard-step1.png)

| 字段 | 必填 | 说明 |
|------|------|------|
| Service Name | ✅ | 服务的显示名称，如 `api-gateway-https` |
| Project | — | 关联的项目 |
| Monitoring | — | 是否启用监控（开关，默认开启） |
| Alerts | — | 是否启用告警通知（开关） |
| Risk Level | — | 风险级别：Low / Medium / High / Critical |

**Step 2 — 主机与检查配置**

![添加服务向导-步骤2](images/ch05-add-service-wizard-step2.png)

- 从下拉菜单选择已有主机，或点击 **+ New Host** 直接在向导中新建
- 选择检查类型后，页面会动态展示对应的配置字段

### 8.3 健康检查类型配置

**TCP 检查**：
- 配置字段：**Port**（必填）
- 说明：验证目标主机指定端口是否开放并可接受连接

**HTTP 检查**：
- 配置字段：**Port**、**Path**（如 `/health`）、**Expected Status Code**（如 `200`）、**Timeout**
- 说明：发送 HTTP GET 请求，验证响应状态码

**HTTPS 检查**：
- 配置字段：同 HTTP，另支持 SSL 证书验证
- 说明：通过 HTTPS 协议检查，可同时验证 SSL/TLS 证书有效性

**Script 脚本检查**：

| 字段 | 说明 |
|------|------|
| Script Type | 脚本类型：bash / python / powershell / nodejs |
| Script Content | 脚本内容（多行编辑器） |
| Expected Exit Code | 期望的退出码，默认为 `0`（表示成功） |

示例 — 检查磁盘使用率：

```bash
#!/bin/bash
USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')
[ "$USAGE" -lt 90 ] && exit 0 || exit 1
```

**File Monitor 文件监控**：

| 字段 | 说明 |
|------|------|
| Mode | **Single File**（单文件）或 **Folder + Pattern**（目录+匹配模式） |
| File Path | 单文件模式下的文件完整路径 |
| Directory Path | 目录模式下的目录路径 |
| Filename Pattern | 文件名匹配模式，支持 glob（如 `*.log`）和正则（如 `app-.*\.log`） |

> 💡 **高级选项**：创建服务后通过**编辑服务**可配置完整的文件检查参数：
> - **Check Type**：`exists`（存在性）/ `size`（大小）/ `modified`（修改时间）/ `content`（内容匹配）
> - **Max Age**：文件的最大允许年龄（支持分钟/小时/天三种单位）
> - **Content Pattern**：内容匹配的正则表达式
> - **Min Size**：文件最小大小（KB）
> - **Freshness Window**：文件新鲜度窗口期（天）
> - **Use Latest Matched File**：目录模式下自动选择最新匹配文件

**数据库检查（即将推出）**：

向导的检查类型选择器中还展示了即将推出的数据库检查类型，目前标记为 Coming Soon 尚不可用：

| 类型 | 说明 |
|------|------|
| 🐬 MySQL Check | MySQL 数据库连通性检查 |
| 🐘 PostgreSQL Check | PostgreSQL 数据库连通性检查 |
| 📮 Redis Check | Redis 缓存服务检查 |
| 🍃 MongoDB Check | MongoDB 数据库连通性检查 |

### 8.4 告警阈值配置

| 字段 | 范围 | 说明 |
|------|------|------|
| **Warning Threshold** | 1-30 | 连续检查失败达到该次数后，服务状态变为 WARNING |
| **Error Threshold** | 1-50 | 连续检查失败达到该次数后，服务状态变为 DOWN |
| **Alert Trigger Threshold** | 1-10 | 连续失败达到该次数后才发送告警通知，避免网络抖动产生误报 |

> 示例：Warning = 2，Error = 5，Alert Trigger = 3 → 连续失败 2 次进入警告，连续失败 5 次标记为 DOWN，连续失败 3 次后发送告警通知

### 8.5 依赖配置

在服务创建或编辑时，可在 **Dependencies** 区域添加服务间的依赖关系：
1. 点击 **Add Dependency**
2. 搜索并选择被依赖的服务
3. 设置**依赖类型**（在系统设置中可自定义）和**风险级别**
4. 依赖关系将反映在[依赖拓扑图](#第十章依赖拓扑图)中

### 8.6 编辑服务

点击服务列表中的 **Edit** 按钮，打开完整的服务编辑表单：

![服务编辑表单](images/ch05-service-edit-form.png)

编辑表单包含比添加向导更全面的配置选项，包括：
- 完整的检查参数配置
- 调度配置（详见 [第九章](#第九章调度配置)）
- 告警自定义内容：
  - **Service Impact Description**：描述服务故障时的影响
  - **Custom Alert Template**：自定义告警消息模板，支持变量：`{service_name}`, `{host}`, `{port}`, `{risk_level}`, `{down_time}`, `{impact_description}`

### 8.7 批量操作

在服务列表页面，勾选多个服务后，顶部会出现批量操作工具栏：

| 操作 | 说明 |

| **Delete** | 批量删除选中服务 |

### 8.8 导入 / 导出服务配置

服务列表页面提供配置的导入导出功能：

### 8.9 手动触发健康检查

### 8.10 服务状态说明

## 第九章：调度配置

每个服务支持独立的检查调度配置，可覆盖系统全局默认间隔。

### 9.1 两种调度方式

#### 固定间隔（Fixed Interval）

最简单直接的调度方式，每隔固定时长执行一次检查：

| 配置项 | 说明 |
|--------|------|
| 间隔时长 | 支持快捷选项（30s / 1min / 5min / 10min / 30min / 1h） |
| 单位切换 | 可在秒（Seconds）、分钟（Minutes）、小时（Hours）之间切换输入 |

示例：设置为 300 秒 → 每 5 分钟检查一次。

#### 时间规则（Time Rules）

通过添加多条时间规则，实现在不同时间段使用不同间隔进行检查：

| 配置项 | 说明 |
|--------|------|
| **星期** | 选择规则生效的星期几（可多选） |
| **时间范围** | 设置每天的起止时间（如 09:00 - 18:00） |
| **检查间隔** | 该时间段内的检查频率（秒） |

**典型场景**：

| 时间段 | 星期 | 间隔 | 目的 |
|--------|------|------|------|
| 09:00 - 18:00 | 周一至周五 | 60s | 工作时间高频检查 |
| 18:00 - 09:00 | 周一至周五 | 600s | 夜间低频检查，减少噪音 |
| 全天 | 周六、周日 | 1800s | 周末值班低频检查 |

> 💡 当时间规则列表为空时，系统自动切回固定间隔模式。

### 9.2 执行预览

调度配置面板底部提供 **Execution Preview（执行预览）** 功能，实时展示调度效果：

| 信息项 | 说明 |
|--------|------|
| **Current Time** | 当前系统时间 |
| **Current Range** | 当前匹配的时间规则名称 |
| **Current Interval** | 当前使用的检查间隔 |
| **Next Run** | 下一次预计执行检查的时间 |

预览区域还包含 **Execution Plan (Next 24 Hours)** 时间线，以可视化方式展示未来 24 小时内各时间段的检查计划和检查间隔。

### 9.3 配置入口

| 入口 | 说明 |
|------|------|
| 服务编辑表单 → 调度配置面板 | 为单个服务独立配置调度 |
| 系统设置 → Default Check Interval | 设定新服务的默认固定间隔 |

## 第十六章：健康检查类型参考

### 16.1 本地检查类型

以下检查类型从监控服务器直接发起，不经过 SSH：

| 检查类型 | 适用主机 | 主要配置参数 | 说明 |
|---------|---------|------------|------|
| **TCP** | Local / SSH | 主机、端口 | 验证端口是否开放并可接受连接 |
| **HTTP** | Local / SSH | 主机、端口、路径、期望状态码、超时 | 发送 HTTP GET 请求验证响应 |
| **HTTPS** | Local / SSH | 同 HTTP，含 SSL 验证 | 支持 TLS/SSL 证书有效性检查 |
| **Script** | Local / SSH | 脚本类型、脚本内容、期望退出码 | 在监控服务器本地执行自定义脚本 |
| **File Monitor** | 需配置 SSH | 模式、路径、检查类型、最大年龄等 | 通过 SSH 在目标主机执行文件检查 |
| **Log Monitor** | 需配置 SSH | 模式、路径、内容匹配模式 | 通过 SSH 监控日志文件内容变化 |

### 16.2 SSH 远程检查类型

以下检查类型通过 SSH 连接到目标主机后，在目标主机上执行（主机须配置 SSH 连接）：

| 检查类型 | 主要配置参数 | 说明 |
|---------|------------|------|
| **file-exists** | 文件路径 | 检查目标主机上指定文件是否存在 |
| **file-age** | 文件路径、最大年龄（秒） | 检查文件修改时间，确保文件足够新鲜 |
| **command** | 命令内容、期望输出匹配模式 | 在目标主机执行任意命令，验证退出码或输出内容 |

### 16.3 混合模式检查

对于 HTTP/HTTPS 和 TCP 检查，支持两种访问模式：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **直接访问** | 监控服务器直接连接到目标地址 | 目标服务对监控网络直接可达 |
| **SSH 代理** | 通过 SSH 连接到主机，在主机内网中执行检查 | 目标服务在防火墙内，仅对主机本机可达 |

## 第十七章：典型使用场景

### 场景一：监控 Web 应用（HTTPS 检查）

**目标**：检查生产环境 Web 服务是否正常响应

**配置步骤**：
1. 创建项目 "Production-Web"
2. 添加主机（IP：`10.0.1.10`，连接类型：Local）
3. 添加服务：
   - Check Type: **HTTPS**
   - Port: `443`
   - Path: `/health`
   - Expected Status Code: `200`
   - Warning Threshold: `2`，Error Threshold: `5`

---

### 场景二：监控数据库服务（TCP 检查）

**目标**：检查数据库端口是否开放，确认服务存活

**配置步骤**：
1. 添加主机（数据库服务器，可为 Local 或 SSH 模式）
2. 添加服务：
   - Check Type: **TCP**
   - Port: `3306`（MySQL）或 `5432`（PostgreSQL）或 `1433`（MSSQL）

---

### 场景三：监控定时任务输出（File Age 检查）

**目标**：确保每小时的数据处理任务正常产出文件（文件不超过 1 小时未更新）

**配置步骤**：
1. 添加主机（任务服务器，SSH 连接）
2. 添加服务：
   - Check Type: **File Monitor**
   - Mode: Single File
   - File Path: `/var/data/job-output/result.csv`
   - Check Type（高级选项）: `modified`
   - Max Age: `3600`（秒）

若文件超过 1 小时未更新，服务状态将变为 WARNING/DOWN。

---

### 场景四：监控磁盘使用率（Script 检查）

**目标**：检查服务器磁盘使用率是否低于 90%

**配置步骤**：
1. 添加主机（目标服务器，SSH 连接）
2. 添加服务：
   - Check Type: **Script**
   - Script Type: `bash`
   - Expected Exit Code: `0`
   - 脚本内容：

```bash
#!/bin/bash
USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')
if [ "$USAGE" -lt 90 ]; then
  echo "Disk usage: ${USAGE}% - OK"
  exit 0
else
  echo "Disk usage: ${USAGE}% - CRITICAL"
  exit 1
fi
```

### 场景五：跨环境服务依赖影响分析

**目标**：微服务架构下，某服务故障时快速识别受影响的上游服务

**配置步骤**：
1. 在各服务的编辑页面 **Dependencies** 区域，配置服务间的依赖关系
2. 进入**依赖拓扑图**（`/graph`）并开启 **Impact Analysis**
3. 当某服务变为 DOWN 状态时，系统自动高亮并列出所有受影响的下游服务

## 第十八章：常见问题与故障排查

### Q1：无法登录系统

**可能原因**：
- 账号或密码错误
- 后端服务未启动

**解决方案**：
1. 确认使用正确的账号密码（默认：`admin` / `admin123`）
2. 访问 `http://localhost:3000/health` 确认后端服务是否正常运行
3. 查看后端日志排查具体错误

---

### Q2：服务状态一直显示 UNKNOWN

**可能原因**：
- 服务的 **Monitoring** 开关未开启
- 尚未到达下一个调度周期
- 后端健康检查服务异常

**解决方案**：
1. 进入服务编辑，确认 **Monitoring** 开关已开启（蓝色）
2. 点击服务行的手动执行按钮（▶）立即触发一次检查
3. 检查后端服务是否正常运行

---

### Q3：SSH 连接测试失败

**可能原因**：
- IP / 端口配置错误
- 用户名或密码/私钥不正确
- 目标服务器防火墙阻止 SSH 访问
- SSH 服务未在目标主机上运行

**解决方案**：
1. 核对主机 IP 和 SSH 端口（默认 22）
2. 确认用户名和认证凭据（密码或私钥）正确
3. 检查服务器防火墙是否开放 22 端口
4. 从本地命令行手动验证：`ssh user@host -p port`

---

### Q4：告警通知未收到

**可能原因**：
- 告警渠道配置错误（SMTP / Webhook URL）
- 服务的 **Alerts** 开关未开启
- 邮件被过滤到垃圾邮件箱

**解决方案**：
1. 在[告警配置页面](#第十一章告警配置)使用 **Test Email** / **Test Teams** 验证配置
2. 确认触发告警的服务已开启 **Alerts** 开关
3. 检查收件人的垃圾邮件文件夹

---

### Q5：端口被占用（启动失败）

**错误信息**：`EADDRINUSE: address already in use :::3000`

**解决方案**：

```powershell
# Windows — 查找并结束占用 3000 端口的进程
netstat -ano | findstr :3000
taskkill /PID <上方显示的PID> /F
```

```bash
# Linux / macOS
lsof -i :3000
kill -9 <PID>
```

### Q6：依赖图加载缓慢或无法显示

**可能原因**：
- 未选择项目（必须先在左侧菜单选择项目）
- 单个项目服务数量过多

**解决方案**：
1. 确认左侧菜单已选择一个具体项目
2. 若服务数量超过 50+，考虑拆分项目以减少单图节点数
3. 刷新页面后重试

### Q7：导入服务配置失败

**可能原因**：
- JSON 文件格式不正确（非系统导出格式）
- 导入的服务关联了不存在的主机或项目

**解决方案**：

3. 查看浏览器控制台或后端日志中的具体错误信息

### Q8：一台主机上监控服务过多，会导致目标服务器 SSH 线程飙升吗？

系统内置了 SSH 连接池来控制连接数，但实际连接数取决于配置方式。

**连接池机制**：

1. **每池硬限制 3 连接**：连接池按 `host:port@username` 唯一键划分，每个池最多 3 个并发连接。超出的请求进入优先级队列排队等待。
2. **连接复用**：SSH 连接执行完检查命令后立即归还池中，下一个排队请求直接复用，无需重新握手。
3. **调度错峰**：SSH 主机的首次检查间隔 10 秒依次启动，避免启动时的连接风暴。
4. **空闲自动回收**：空闲超过 1 分钟的连接自动关闭（每 30 秒清理一次），不会长期占用目标服务器资源。

**为什么可能看到超过 3 个连接？**

连接池的隔离单元是 **凭据组合**（`host:port@username`），而非物理服务器。以下场景会在同一台服务器上创建多个独立连接池：

| 场景 | 连接池数 | 最大连接数 |
|------|---------|-----------|
| 同一服务器、同一用户名 | 1 个池 | 3 |
| 同一服务器、2 个不同用户名 | 2 个池 | 6 |
| 同一服务器、3 个不同用户名 | 3 个池 | 9 |
| 经同一跳板机的 N 台目标主机 | N 个池 | N × 3（跳板机侧） |

**常见的多连接池情况**：
- **主机心跳**使用 hosts 表的凭据（如 `admin@server`），**SSH 类型服务检查**使用 security_configs 表的凭据（如 `deploy@server`）—— 用户名不同则产生两个独立池（最多 6 连接）。
- 多台主机经过**同一跳板机**时，跳板机上会累积所有目标主机的代理连接。

**优化建议**：
- 尽量对同一台服务器使用**统一的 SSH 用户名**，减少连接池数量。
- 对跳板机后的大量主机，关注跳板机的 SSH 最大连接数配置（`MaxSessions`、`MaxStartups`）。
